# CONTEXTO MEJORADO — FLUJO DE ACCESO, PRIMER INGRESO Y CONFIGURACIÓN INICIAL DEL SISTEMA

# FILOSOFÍA GENERAL DEL ACCESO

El sistema NO permite auto-registro público.

Toda cuenta debe ser creada previamente por:

* Superadmin
* Administrador autorizado

El sistema está diseñado para entornos empresariales internos donde:

* los usuarios ya pertenecen a la organización,
* los accesos son controlados,
* y la identidad organizacional ya existe antes del ingreso.

Por lo tanto:

* NO existe botón de “Registrarse”
* NO existe onboarding público
* NO existe creación autónoma de cuentas

---

# SISTEMA DE AUTENTICACIÓN

## Métodos de acceso permitidos

El sistema permite únicamente:

### 1. Login tradicional

Mediante:

* Usuario/email
* Contraseña

---

### 2. OAuth con Google

Permitido únicamente si:

* el usuario YA existe en la base de datos,
* y su cuenta fue creada previamente por un administrador.

## Regla importante

OAuth NO crea usuarios automáticamente.

Si el correo no existe:

→ acceso denegado.

---

# SEGURIDAD DEL LOGIN

## Protección anti-spam y fuerza bruta

El sistema implementa límites de intentos fallidos.

### Reglas iniciales

* Máximo 3 intentos fallidos consecutivos
* Bloqueo temporal automático
* Tiempo configurable (ejemplo: 15 minutos)
* Registro en auditoría de seguridad

---

## Mensajes del sistema

El sistema debe evitar revelar información sensible.

Ejemplo correcto:

```txt
Usuario o contraseña incorrectos
```

NO debe indicar:

* si el usuario existe,
* si el email es válido,
* ni qué dato falló.

---

# FLUJOS PRINCIPALES DE INGRESO

Existen 2 escenarios operativos principales:

---

# ESCENARIO 1 — PRIMER INGRESO AL SISTEMA

Ocurre cuando:

* la cuenta fue creada por un admin/superadmin,
* pero el usuario aún no completa su configuración inicial.

El sistema detecta automáticamente:

```txt
profile_completed = false
```

y redirige al flujo onboarding inicial.

---

# PRIMER INGRESO — USUARIO ESTÁNDAR

## Objetivo

Completar información organizacional y asegurar la cuenta.

---

# CONFIGURACIÓN INICIAL DEL USUARIO

El usuario debe completar:

## Información personal

* Nombre
* Username
* Teléfono
* Dirección
* País
* Departamento/estado
* Ciudad

---

## Información organizacional

Estos datos NO son libres.

Deben provenir dinámicamente desde la configuración del sistema creada por el superadmin.

Ejemplos:

* Sede
* Área
* Departamento
* Cargo
* División
* Tipo de usuario

Todos estos campos deben utilizar:

* dropdowns,
* listas dinámicas,
* búsquedas,
* selects relacionados.

---

# CONTRASEÑA INICIAL

Toda cuenta creada por administradores tendrá:

```txt
Ticket2026!
```

como contraseña temporal inicial.

---

# CAMBIO OBLIGATORIO DE CONTRASEÑA

En el primer ingreso:

* el sistema obliga a cambiar contraseña,
* antes de permitir acceso operativo completo.

La contraseña temporal NO puede reutilizarse.

---

# CONFIGURACIÓN DE SEGURIDAD

Durante el onboarding inicial el usuario podrá configurar:

## 2FA interno del sistema

Basado en:

* El codigo de verificacion ya puesto en este sistema con smtp con resend

El usuario podrá:

* activar,
* desactivar,
* o posponer la activación (según política definida).

---

# OBJETIVO DEL ONBOARDING

El flujo inicial busca:

* completar identidad organizacional,
* asegurar cuentas,
* preparar automatizaciones futuras,
* permitir trazabilidad,
* habilitar permisos contextuales,
* alimentar reportes y métricas.

---

# PRIMER INGRESO — SUPERADMIN

## Condición especial

Si el sistema detecta que NO existe configuración organizacional global:

```txt
company_initialized = false
```

entonces el primer superadmin NO entra al dashboard operativo.

---

Este flujo reutiliza visualmente la experiencia de onboarding de usuario, pero funcionalmente configura TODO el sistema.

* Single Tenant

# CONSISTENCIA ENTRE MÓDULOS

Todos los módulos deben mantener:

* misma lógica visual,
* mismas bases UX,
* estructura operacional consistente,
* navegación coherente.

Esto reduce:

* curva de aprendizaje,
* fricción operativa,
* desorientación del usuario.

---

# ESTRUCTURA BASE DE TICKETS EN TODOS LOS MÓDULOS

Todos los módulos, incluso los personalizados, deben heredar una estructura base de trabajo tipo tickets.

Esto significa que cada módulo debe soportar conceptualmente:

* solicitudes,
* estados,
* responsables,
* historial,
* comentarios,
* trazabilidad,
* prioridades,
* flujo operativo.

La referencia principal para esta arquitectura será:

```txt
Helpdesk
```

---

# RUTAS INTERNAS DE LOS MÓDULOS

Las rutas internas deben seguir estructura contextual.

Ejemplo:

```txt
/modules/helpdesk/trash
/modules/inventory/reports
/modules/administrative-management/users
```

Esto garantiza:

* modularidad,
* separación lógica,
* reutilización de componentes,
* permisos contextuales.

---

# GESTIÓN ADMINISTRATIVA — SISTEMA DE SOLICITUDES

# CREACIÓN DE SOLICITUDES

Las solicitudes se crean mediante modal contextual.

---

# TIPOS DE SOLICITUDES DISPONIBLES

Inicialmente el sistema soportará:

* Cambio de rol
* Acceso a módulo
* Ajustes de permisos
* Problema de cuenta
* Reactivación
* Revocación de acceso
* Traslado de usuarios
* Problemas técnicos
* Corrección de datos
* Tareas internas
* Otros

---

# TIPO “OTROS”

Cuando se selecciona:

```txt
Otros
```

el sistema debe mostrar explicación contextual indicando:

* qué casos aplican,
* limitaciones,
* alcance de la solicitud.

---

# ESTRUCTURA DEL FORMULARIO

La solicitud debe permitir seleccionar:

## Tipo de solicitud

Desde dropdown dinámico.

---

## Módulo relacionado

El usuario podrá seleccionar:

* módulo global,
* Helpdesk,
* Inventario,
* Gestión Administrativa,
* módulos personalizados existentes en DB.

Todo debe provenir dinámicamente desde configuración real del sistema.

---

# PRIORIDAD AUTOMÁTICA

La prioridad NO será manual en la mayoría de casos.

El sistema calculará automáticamente:

* prioridad,
* urgencia,
* impacto,

según:

* tipo de solicitud,
* rol del usuario,
* contexto organizacional,
* reglas definidas.

---

# CAMPOS PRINCIPALES

La solicitud incluye:

* título,
* descripción,
* contexto,
* módulo relacionado,
* trazabilidad.

---

# SISTEMA DE USUARIOS DENTRO DE LOS MÓDULOS

# VISIBILIDAD ESPECIAL DE USUARIOS

Los módulos:

* Gestión Administrativa
* Inventario

requieren acceso global a usuarios organizacionales.

Por esta razón:

NO necesitan solicitar permisos para visualizar usuarios.

---

# VISIBILIDAD EN OTROS MÓDULOS

En módulos distintos:

el acceso sí requiere autorización previa.

---

# FLUJO DE APROBACIÓN DE SOLICITUDES

Cuando un usuario solicita:

* ingreso a módulo,
* cambio de rol,
* permisos,
* ajustes operativos,

la solicitud se envía automáticamente al:

```txt
admin/aprobador del módulo correspondiente
```

---

# ESCALAMIENTO AUTOMÁTICO

Si el módulo no posee admin asignado:

la solicitud escala automáticamente al:

```txt
superadmin
```

---

# NOTIFICACIONES OPERATIVAS

El admin del módulo recibe la solicitud mediante:

## 1. Centro de notificaciones

Header global.

---

## 2. Bandeja interna de Gestión Administrativa

Vista operativa de solicitudes pendientes.

---

# REGLA OPERATIVA IMPORTANTE

Aunque la solicitud se gestione desde:

```txt
Gestión Administrativa
```

los cambios reales deben ejecutarse desde el módulo correspondiente.

---

# EJEMPLO

Si un usuario solicita:

```txt
Cambio de rol en Inventario
```

entonces:

* la solicitud aparece en Gestión Administrativa,
* el admin de Inventario la revisa,
* pero el cambio real se ejecuta desde:

```txt
/modules/inventory
```

porque ese módulo controla:

* sus roles,
* permisos,
* estructura interna.

---

# PRINCIPIO DE RESPONSABILIDAD MODULAR

Cada módulo es responsable de administrar:

* sus roles,
* usuarios internos,
* permisos contextuales,
* configuraciones operativas.

Gestión Administrativa únicamente:

* centraliza solicitudes,
* enruta procesos,
* mantiene trazabilidad,
* coordina aprobaciones.

---

# REESCALAMIENTO DE SOLICITUDES

Los admins de módulo pueden:

* escalar solicitudes,
* reenviarlas,
* solicitar revisión superior.

Especialmente cuando:

* la acción supera sus permisos,
* afecta múltiples módulos,
* involucra seguridad,
* requiere aprobación organizacional.

---

# TRAZABILIDAD OBLIGATORIA

Toda solicitud debe conservar:

* historial completo,
* responsables,
* cambios realizados,
* aprobaciones,
* rechazos,
* escalamientos,
* comentarios,
* timestamps,
* auditoría operativa.
