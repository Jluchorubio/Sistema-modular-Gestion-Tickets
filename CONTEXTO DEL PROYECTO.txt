# VISIÓN GENERAL

NEXO ITSM es una plataforma modular empresarial que centraliza:

- Gestión de incidentes y solicitudes técnicas (Helpdesk)
- Gestión de activos tecnológicos (Inventario CMDB)
- Solicitudes administrativas internas (Gestión Administrativa)
- Notificaciones en tiempo real
- Calendario operacional
- Reportes y KPIs
- Auditoría completa
- Control de roles y permisos

Es SINGLE TENANT: una empresa, una instancia, una organización.

La arquitectura modular permite que cada módulo opere autónomamente pero comparta:
- usuarios y autenticación
- permisos granulares
- notificaciones
- calendario
- motor organizacional
- reportes

# STACK TECNOLÓGICO (ACTUAL)

## Backend
- NestJS 11 + TypeScript
- PostgreSQL 16 en Railway (15 schemas, schema v7.0)
- JWT propio (access 15 min / refresh 7 días)
- OAuth Google (passport-google-oauth20)
- TOTP 2FA (speakeasy)
- Emails transaccionales (Resend)
- Socket.IO para notificaciones en tiempo real
- EventEmitter2 como bus de eventos interno
- Schedulers @nestjs/schedule (SLA auto-escalation cada 30 min)
- Rate limiting (100 req/60s por IP)

## Frontend
- Next.js 15 App Router + TypeScript
- CSS Modules (estilos por componente — NO usa Tailwind)
- Zustand (estado global: auth, UI, permisos)
- TanStack Query / React Query (fetching, caché, invalidación)
- Socket.IO Client (notificaciones en tiempo real)
- Recharts (gráficos de reportes)
- Lucide React (iconografía)

## Infraestructura
- Docker + Docker Compose
- Nginx (proxy reverso)
- GitHub Actions → deploy automático vía SSH al servidor
- Dominio producción: nexo.proyectoscampus.top

# AUTENTICACIÓN — FLUJO COMPLETO

## Login
Ruta: /login

El usuario ingresa con:
- Email + contraseña
- O hace clic en "Login con Google" (OAuth 2.0)

Seguridad:
- 3 intentos fallidos → bloqueo temporal progresivo
- Mensajes genéricos: nunca indica qué campo falló
- Rate limiting 100 req/60s por IP

Si el usuario tiene 2FA activado:
1. Login retorna { requires_mfa: true, mfa_token }
2. Se muestra pantalla de TOTP
3. El usuario ingresa el código de 6 dígitos de Google Authenticator
4. Backend valida y emite tokens reales

## Primer ingreso de usuario normal
La contraseña temporal por defecto es: Ticket2026!

Al ingresar por primera vez:
1. El sistema detecta force_password_change = true
2. Redirige a /complete-profile
3. El usuario completa 3 pasos:
   - Información personal (nombre, teléfono, fecha nacimiento, idioma, zona horaria)
   - Datos organizacionales: selecciona departamento → área (cascading desde org.nodes de la empresa)
   - Seguridad: cambia contraseña temporal por una nueva (mín. 8 chars, 1 mayúscula, 1 minúscula, 1 número)

## Primer ingreso del superadmin
Si no existe configuración empresarial → Setup Wizard.
Configura: empresa, branding, estructura organizacional, módulos base.

## Recuperación de contraseña
Ruta: /login → "Olvidé mi contraseña"
1. Ingresa email
2. Recibe enlace con token de recuperación (expira en 1h)
3. Ingresa nueva contraseña

## NO existe registro público
Los usuarios SOLO son creados por superadmin o admin_modulo desde /users.

# JERARQUÍA DE ROLES

## Roles globales
- superadmin: bypass total, config global, todos los módulos, auditoría, usuarios
- usuario: rol base de cualquier empleado

## Roles por módulo (se asignan por módulo específico)
- admin_modulo: gestión completa del módulo (config, usuarios, reportes, flujos)
- jefe_tecnico: supervisión operativa (reasignar, escalar, métricas, SLA)
- tecnico: trabajar tickets asignados, cambiar estados, comentarios internos
- usuario: crear tickets/solicitudes, ver los suyos, aprobar resolución

Un mismo usuario puede tener roles distintos en módulos distintos.
Ejemplo: tecnico en Helpdesk, usuario en Inventario, jefe_tecnico en otro módulo.

## RBAC granular
60 permisos definidos en DB, agrupados por recurso:
- tickets: create, view, view_all, edit, assign, close, delete, manage
- inventory: view, create, edit, delete, manage
- requests: create, view, view_all, review, manage
- users, roles, config, reports, audit, trash: varios permisos cada uno

Los permisos se asignan por usuario/módulo desde /roles o /config.
El superadmin bypassa todos los checks directamente.

## Máquina de estados del ticket (FSM)

Estados posibles:
- Abierto (creación)
- En proceso (tomado por técnico)
- En espera (esperando info del usuario)
- Realizado (técnico marca como hecho)
- Cerrado (usuario aprueba o timeout)
- Reproceso (usuario rechaza la resolución)
- Resuelto (cierre directo por técnico/jefe)
- Rechazado (descartado por jefe/admin)

Transiciones con control de rol:
- Tomar (abierto → en_proceso): tecnico, jefe_tecnico, admin_modulo
- Solicitar info (en_proceso → en_espera): tecnico, jefe_tecnico, admin_modulo
- Reanudar (en_espera → en_proceso): tecnico, jefe_tecnico, admin_modulo
- Marcar realizado (en_proceso → realizado): tecnico, jefe_tecnico, admin_modulo
- Aprobar y cerrar (realizado → cerrado): usuario, jefe_tecnico, admin_modulo
- Rechazar solución (realizado → reproceso): usuario, jefe_tecnico, admin_modulo
- Retomar reproceso (reproceso → en_proceso): tecnico, jefe_tecnico, admin_modulo
- Rechazar ticket (en_proceso → rechazado): jefe_tecnico, admin_modulo
- Reabrir (resuelto → en_proceso): jefe_tecnico, admin_modulo

Superadmin bypassa todos los checks.

---

## Flujo de validación y calificación

1. Técnico marca como "Realizado"
2. Usuario recibe notificación → ve banner de validación con firma digital
3. Usuario elige:
   - APROBAR: firma → ticket a estado final → sistema pide calificación (1-5 estrellas + comentario)
   - RECHAZAR: justificación → ticket a "Reproceso" (solo 1 reproceso permitido)
4. Si el usuario no responde en X horas → el sistema cierra automáticamente (sin calificación)
5. Si el reproceso falla → escalamiento automático

---

## SLA del Helpdesk

El motor SLA controla prioridad, orden de atención y escalamiento.

- El reloj inicia cuando el ticket pasa a "En proceso"
- Se pausa automáticamente en "En espera"
- Al salir de "En espera" el deadline se extiende
- Se cierra como "met" (cumplido) o "breached" (vencido) al llegar a estado final
- El scheduler corre cada 30 minutos: escala automáticamente tickets vencidos

Los SLA se configuran por categoría y prioridad en /config → Configuración del módulo.
Se respetan horarios laborales y días feriados configurados en Config Global.

# MÓDULO GESTIÓN ADMINISTRATIVA — FLUJO COMPLETO

Ruta base: /requests

Hub de solicitudes internas no técnicas.

## Tipos de solicitudes disponibles
- Cambio de rol
- Solicitud de acceso a módulo
- Corrección de datos personales
- Cambio de sede o área
- Asignación de permiso
- Problema de cuenta
- Reactivación de cuenta
- Tarea administrativa
- Otro

## Flujo de una solicitud

1. El usuario crea la solicitud: tipo, descripción, documentos adjuntos
2. El sistema la enruta al admin_modulo correspondiente
   - Si el módulo no tiene admin → va automáticamente al superadmin
3. El admin revisa → puede:
   - TOMAR: asignarse la solicitud
   - APROBAR con comentario
   - RECHAZAR con justificación
   - ESCALAR al superadmin
4. Cada acción queda auditada
5. El usuario recibe notificación de cada cambio de estado

## Etapas del flujo
pending → taken → in_progress → under_review → approved / rejected / cancelled

## Vistas por rol

Usuario: ve sus solicitudes, puede crearlas, consultar estado, adjuntar documentos.
Admin / Superadmin: ve todas las solicitudes, gestión completa, métricas globales.

# PAPELERA — /trash

Soft-delete unificado para todo el sistema.

Reglas:
- Los elementos eliminados se guardan 90 días antes del hard delete
- El sistema notifica progresivamente cuando un elemento está próximo a vencer
- Para eliminar definitivamente antes de los 90 días: re-autenticación (contraseña)
- Bulk select disponible (eliminar varios a la vez)

Acceso:
- superadmin: ve todo lo global
- admin_modulo: solo lo de su módulo

EXCEPCIÓN CRÍTICA — Tickets:
Los tickets NUNCA van a papelera.
Son registros permanentes del sistema.
Solo se pueden Cancelar o Archivar.
Esto preserva trazabilidad, SLA histórico, métricas y auditoría.

# PWA (PROGRESSIVE WEB APP)
