🔐 Flujo de Autenticación del Sistema
Objetivo

Permitir el acceso seguro al sistema mediante dos métodos de autenticación:

Inicio de sesión tradicional (usuario + contraseña)
Inicio de sesión con Google OAuth

El acceso solo será permitido a usuarios previamente registrados en la base de datos del sistema.

1. Inicio de Sesión Tradicional
Descripción General

El usuario podrá autenticarse utilizando:

Usuario o correo electrónico
Contraseña
Flujo de Autenticación
Paso 1 — Ingreso de Credenciales

El usuario ingresa:

Usuario/correo
Contraseña

y presiona el botón “Iniciar sesión” o la tecla Enter.

Paso 2 — Validación de Existencia

El sistema debe verificar:

Que el usuario exista en la base de datos.
Que la cuenta esté activa.
Que tenga permisos de acceso.
Resultado esperado
Si el usuario NO existe:

Mostrar mensaje flotante:

“Usuario no existente.”

Si existe:
Continuar al siguiente paso.
Paso 3 — Validación de Contraseña

El sistema valida la contraseña ingresada.

Política de Intentos Fallidos

El usuario tendrá un máximo de:

3 intentos fallidos consecutivos.
Si supera el límite:

El sistema debe:

Bloquear temporalmente el inicio de sesión.

Mostrar mensaje indicando:

“Demasiados intentos fallidos. Intente nuevamente más tarde.”

Consideraciones técnicas

El bloqueo puede implementarse mediante:

Rate limiting por IP
Rate limiting por usuario
Tiempo de espera progresivo
Registro de intentos fallidos en logs de auditoría
Paso 4 — Autenticación en Dos Factores (2FA)

Si las credenciales son correctas:

El sistema generará un código temporal de verificación.
El código será enviado al correo electrónico del usuario.
El usuario deberá ingresar dicho código para completar el acceso.
Validación del Código 2FA

El sistema debe validar:

Que el código sea correcto.
Que no haya expirado.
Que pertenezca al usuario autenticado.
Resultado esperado
Si el código es inválido:

Mostrar mensaje:

“Código incorrecto o expirado.”

Si es válido:
Permitir acceso al sistema.
2. Inicio de Sesión con Google
Descripción General

El usuario podrá iniciar sesión mediante Google OAuth.

⚠️ Importante:
El sistema NO debe registrar automáticamente usuarios nuevos desde Google.

Solo podrán acceder usuarios previamente existentes en la base de datos.

Flujo de Autenticación con Google
Paso 1 — Selección de Google Login

El usuario presiona:

“Iniciar sesión con Google”

Paso 2 — Autenticación con Google

Google devuelve:

Correo electrónico autenticado
Información básica del usuario
Paso 3 — Validación en Base de Datos

El sistema verifica:

Que el correo exista en la base de datos.
Que el usuario esté habilitado.
Resultado esperado
Caso válido

Si el correo existe:

Permitir acceso al sistema.
Caso inválido

Si el correo NO existe:

El sistema NO debe:

Registrar usuarios automáticamente
Crear cuentas nuevas
Guardar información en base de datos
En su lugar:

Mostrar mensaje flotante:

“Usuario no existente.”

y cancelar el acceso.

3. Reglas Generales de Seguridad
Requisitos Obligatorios
Seguridad de contraseñas
Contraseñas encriptadas con bcrypt o Argon2.
Nunca almacenar contraseñas en texto plano.
Seguridad de sesión
Uso de JWT o sesiones seguras.
Tokens con expiración.
Refresh tokens opcionales.
Auditoría

Registrar:

Intentos fallidos
Bloqueos temporales
Inicios de sesión exitosos
Uso de autenticación Google
Errores de autenticación
Protección adicional

Implementar:

Rate limiting
Protección contra fuerza bruta
Validación de IP/dispositivo opcional
CSRF protection
Secure cookies
4. Comportamientos Esperados del Sistema
Escenario	Resultado
Usuario inexistente	Mostrar “Usuario no existente”
Contraseña incorrecta	Consumir intento
3 intentos fallidos	Bloqueo temporal
Código 2FA incorrecto	Denegar acceso
Código expirado	Solicitar nuevo código
Login Google con usuario existente	Acceso permitido
Login Google con usuario no registrado	Denegar acceso sin registrar usuario
5. Restricciones Importantes
El sistema NO debe:
Registrar usuarios automáticamente con Google.
Permitir bypass del 2FA.
Permitir intentos infinitos de login.
Exponer información sensible en errores.
Diferenciar visualmente entre:
usuario inexistente
contraseña incorrecta

(para evitar enumeración de usuarios).

🗑️ Eliminación de Módulos
Política Obligatoria de Eliminación

Cuando el Super Admin elimine un módulo/proyecto, el sistema deberá utilizar obligatoriamente un proceso de confirmación tipo GitHub.

No podrá existir eliminación directa con un solo clic.

Flujo Obligatorio de Eliminación

Antes de eliminar un módulo, el sistema deberá:

Mostrar una advertencia clara sobre el impacto de la eliminación.
Solicitar confirmación manual.
Solicitar que el Super Admin escriba explícitamente:
Su nombre de usuario
El nombre exacto del módulo/proyecto a eliminar

Solo si ambos datos coinciden correctamente:

Se habilitará el botón de eliminación.
♻️ Tipo de Eliminación

La eliminación será mediante:

Soft Delete

Esto significa que el módulo NO será eliminado inmediatamente de la base de datos.

🕒 Tiempo de Retención

Una vez eliminado:

El módulo permanecerá almacenado durante 3 meses.

Durante este período:

El Super Admin podrá recuperar completamente el módulo.
Se deberán conservar:
Tickets
Usuarios
Roles
Configuraciones
SLA
Nomenclaturas
Historial
Archivos
Relaciones internas
Auditorías
Configuraciones personalizadas

La restauración deberá devolver el módulo exactamente al estado previo a su eliminación.

❌ Eliminación Permanente

Si transcurren los 3 meses de retención:

El sistema eliminará automáticamente el módulo de forma definitiva (Hard Delete).

Esta eliminación permanente incluirá:

Datos
Relaciones
Configuraciones
Historial asociado
Archivos relacionados

Después de este proceso:

La recuperación ya no será posible.

# 🔔 🔐 VALIDACIÓN Y CIERRE (REGLA CRÍTICA DEL SISTEMA)

## 🔹 Condición obligatoria de cierre

> Un ticket SOLO se cierra cuando:

* El técnico finaliza
* El usuario valida la solución (firma digital)

## 🔹 Notificación al usuario

El usuario recibe:

* Notificación interna (obligatoria)
* Email (Gmail por defecto)
* WhatsApp (opcional)

Incluye:

* Solución aplicada
* Evidencias
* Token de firma digital

---

## 🔹 Escenarios de cierre

### ✅ Aprobación

* Usuario firma digitalmente
* Estado → ⚪ Cerrado

---

### ⏳ Sin respuesta

* Si no responde en 2 días:
* Estado → ⚪ Cerrado automático

---

### ❌ Rechazo → Reproceso

Condiciones:

* Usuario no está satisfecho
* Debe enviar:

  * Justificación
  * Evidencias

Resultado:

* Estado → 🔴 Reproceso
* Se invalida la solución previa

---

## 🔁 6. Reproceso (controlado)

* El técnico vuelve a trabajar el ticket
* Se repite el flujo de validación

---

## 🔴 Regla crítica

> ❗ Solo se permite **UN reproceso**

---

## 🔺 7. Escalamiento

Si el usuario vuelve a rechazar:

* El ticket se escala

* Se reasigna (Jefe Técnico)

* Prioridad aumenta:

  * Máximo → **Alta (nunca Crítica)**

* Resolución obligatoria
