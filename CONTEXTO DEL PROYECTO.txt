# FILOSOFÍA GENERAL DEL SISTEMA

El sistema es una plataforma empresarial modular single-tenant diseñada para gestión operativa interna. No es solo un sistema de tickets: integra Mesa de Ayuda, Inventario, Gestión Administrativa y módulos dinámicos creados por el superadmin, todos compartiendo la misma infraestructura de usuarios, autenticación, permisos, auditoría y notificaciones.

Cada despliegue pertenece a una sola organización. No existe multiempresa en la misma instancia.

# ACCESO AL SISTEMA

## Regla fundamental

El sistema NO permite auto-registro público. Toda cuenta debe ser creada previamente por un superadmin o administrador autorizado. No existe botón de "Registrarse" en ninguna parte del sistema. El sistema está diseñado para entornos empresariales donde los usuarios ya pertenecen a la organización y su identidad existe antes de ingresar.

---

## Métodos de acceso permitidos

### Login tradicional

El usuario accede mediante email y contraseña. Al ingresar credenciales correctas, el sistema NO abre sesión inmediatamente: siempre solicita verificación de identidad adicional.

### OAuth con Google

Permitido únicamente si el usuario ya existe en la base de datos con cuenta creada previamente por un administrador. OAuth NO crea usuarios automáticamente. Si el correo de Google no existe registrado en el sistema, el acceso es denegado sin excepción.

---

## Flujo de login paso a paso

1. El usuario ingresa email y contraseña en la pantalla de login (`/login`).
2. El sistema verifica las credenciales. Si son incorrectas, responde con mensaje genérico: "Usuario o contraseña incorrectos". El mensaje nunca revela si el email existe o no, ni qué dato falló.
3. Si las credenciales son correctas, el sistema envía un código OTP de 6 dígitos al correo del usuario mediante Resend API.
4. El usuario ingresa el código OTP en la pantalla siguiente. Tiene 10 minutos para usarlo.
5. Si el código es correcto, se crea la sesión completa.
6. Si el código es incorrecto 3 veces consecutivas, el OTP queda quemado y la cuenta se bloquea automáticamente durante 15 minutos. Durante ese tiempo ningún intento de login es posible.

---

## Seguridad anti-fuerza bruta

El sistema implementa bloqueo temporal automático. Los intentos fallidos se registran en auditoría de seguridad. El tiempo de bloqueo es configurable por el superadmin.

---

## Contraseña temporal inicial

Toda cuenta creada por administradores tiene como contraseña temporal:

```
Ticket2026!
```

Esta contraseña es solo para el primer ingreso. El sistema la detecta automáticamente y obliga al cambio antes de permitir acceso operativo.

# ESCENARIOS DE INGRESO

## Escenario 1: Primer ingreso de usuario

Cuando un administrador crea una cuenta nueva y el usuario ingresa por primera vez, el sistema detecta `profile_completed = false` y redirige automáticamente al flujo de onboarding inicial antes de dar acceso al sistema operativo.

Durante el onboarding el usuario completa:

**Información personal**: nombre, apellido, username, teléfono, dirección, país, departamento/estado, ciudad.

**Información organizacional**: estos datos NO son de escritura libre. Provienen de dropdowns dinámicos poblados desde la configuración del sistema creada por el superadmin: sede, área, departamento, cargo. El usuario selecciona desde listas predefinidas.

**Cambio obligatorio de contraseña**: el sistema obliga a reemplazar la contraseña temporal `Ticket2026!`. La contraseña temporal no puede reutilizarse. Hasta que no se cambie, el acceso al sistema está bloqueado.

**Configuración de seguridad (2FA)**: durante el onboarding el usuario puede activar, desactivar o posponer la activación del segundo factor de autenticación según la política definida por la organización.

El onboarding funciona como wizard multi-step con guardado progresivo y validación por secciones. El usuario NO puede operar el sistema hasta completarlo.

---

## Escenario 2: Primer ingreso del superadmin (inicialización del sistema)

Si el sistema detecta que no existe configuración organizacional global (`company_initialized = false`), el primer superadmin NO entra al dashboard operativo. En su lugar, entra a un flujo especial de inicialización empresarial en `/setup`.

Este flujo configura toda la estructura organizacional requerida por el sistema:

**Información corporativa**: nombre de la empresa, logo, branding, dominio corporativo, correos organizacionales.

**Estructura organizacional**: sedes, departamentos, áreas, cargos, jerarquías.

**Configuración operacional**: categorías, prioridades, reglas SLA, roles base, tipos de solicitudes, políticas de acceso.

Este paso es obligatorio antes de que cualquier otro usuario pueda completar su perfil, porque los dropdowns organizacionales del onboarding de usuarios dependen directamente de esta configuración.

---

## Escenario 3: Ingreso cotidiano

Cuando `profile_completed = true` y `company_initialized = true`, el usuario ingresa directamente al sistema operativo después de pasar el login + OTP.

# ESTRUCTURA GENERAL DE LA INTERFAZ

La aplicación se divide en tres zonas principales:

**Header superior global**: visible para todos los usuarios autenticados en todo momento. Contiene logo, acceso al calendario, notificaciones y el dropdown del perfil.

**Sidebar lateral izquierdo**: visible principalmente para superadmin y roles con permisos administrativos globales. Los usuarios estándar tienen una experiencia más limpia sin este sidebar.

**Área central dinámica**: donde se renderizan el dashboard, los módulos, los tickets, los reportes, las configuraciones y todas las vistas operativas.

---

# HEADER GLOBAL

El header está presente en todas las pantallas del sistema una vez autenticado. Contiene cuatro elementos:

## Logo

Ubicado a la izquierda. Muestra el branding de la empresa configurado por el superadmin. Al hacer clic regresa al dashboard principal.

## Acceso al calendario

Un botón contextual en el header despliega el acceso rápido al calendario operativo global. El calendario no vive en el sidebar sino en el header porque todos los roles lo utilizan: es un sistema transversal, no exclusivo de ningún área administrativa.

## Centro de notificaciones

Un ícono de campana con badge numérico indica la cantidad de notificaciones no leídas. Al hacer clic abre un dropdown con la lista de notificaciones recientes. Cada notificación puede marcarse como leída individualmente. Existe también un botón "Marcar todas como leídas". Las notificaciones se generan automáticamente por: transiciones de tickets, aprobaciones o rechazos de solicitudes, invitaciones a colaborar en tickets, alertas de vencimiento SLA y eventos del calendario.

## Dropdown del perfil

Al hacer clic en el avatar del usuario autenticado se despliega un panel inspirado en experiencias modernas tipo Discord/Slack. Contiene las siguientes opciones:

**Perfil**: redirige a `/profile`, la vista completa del perfil del usuario.

**Tema**: permite cambiar entre modo claro, modo oscuro o sincronizado con el sistema operativo.

**Historial de tickets**: acceso rápido a los tickets personales del usuario autenticado.

**Descargar aplicación**: preparado para futura versión PWA/app móvil.

**Cerrar sesión**: finaliza la sesión activa actual y redirige al login.

# DASHBOARD PRINCIPAL (`/dashboard`)

El dashboard es el hub operativo del sistema. Al ingresar, el usuario ve todos los módulos disponibles según sus permisos, representados como cards en una grilla.

## Cards de módulos

Cada módulo está representado por una card operativa. Todas las cards mantienen el mismo tamaño, la misma altura y una grilla uniforme. El texto se trunca si es demasiado largo. Tienen hover state, son responsive y visualmente consistentes.

Cada card muestra:
- Imagen o banner representativo del módulo
- Nombre del módulo
- Descripción breve del módulo
- Estado visual: activo, desactivado, o en mantenimiento

Al hacer clic en una card el usuario entra al módulo correspondiente, siempre que tenga acceso autorizado.

## Menú contextual de módulos (3 puntos)

Cada card tiene un botón de 3 puntos (kebab menu) en la esquina superior derecha. Al hacer clic se despliega un menú con las acciones disponibles según el nivel de permisos del usuario.

### Editar

Abre un modal que permite modificar el nombre del módulo, la descripción, la imagen o banner, y configuraciones visuales básicas del módulo. Los cambios se guardan inmediatamente al confirmar.

### Activar / Desactivar

Permite habilitar o deshabilitar operativamente el módulo. Cuando un módulo está desactivado, los usuarios normales no pueden acceder a él. El módulo puede mostrarse con un estado visual diferente (atenuado o con badge "desactivado") para indicar que no está disponible. El superadmin y admins autorizados siempre pueden ingresar independientemente del estado.

### Modo mantenimiento

Coloca el módulo en estado de mantenimiento. En este estado, únicamente los admins autorizados y el superadmin pueden ingresar al módulo. El objetivo es permitir que el equipo técnico realice cambios críticos como: modificación de reglas SLA, ajuste de workflows, cambios de automatizaciones, modificación de permisos o reestructuración de configuraciones. Mientras el modo mantenimiento está activo, el resto de usuarios ve el módulo restringido. Es diferente a "desactivado": desactivado es operacional, mantenimiento es temporal para cambios internos.

### Eliminar módulo

**Solo disponible para módulos personalizados creados dinámicamente.** Los módulos core del sistema (Helpdesk, Inventario, Gestión Administrativa) NO pueden eliminarse bajo ninguna circunstancia.

Al seleccionar eliminar, el sistema abre un modal de confirmación reforzada que solicita dos campos:
1. Escribir manualmente el nombre exacto del módulo
2. Escribir manualmente el nombre del usuario autenticado

Esto evita eliminaciones accidentales y acciones masivas peligrosas. Solo cuando ambos campos coinciden exactamente el botón de confirmación se habilita.

La eliminación realiza soft-delete: el módulo NO se borra de la base de datos inmediatamente. Pasa automáticamente a la papelera global (`/trash`) donde permanece 90 días antes de la eliminación definitiva automática.

## Módulos existentes siempre presentes (built-in)

El sistema siempre incluye tres módulos que no pueden eliminarse:

**Helpdesk / Mesa de Ayuda**: gestión de tickets de soporte técnico. Ruta: `/helpdesk`.

**Inventario**: gestión de activos y trazabilidad organizacional. Ruta: `/inventory`.

**Gestión Administrativa**: centro de solicitudes organizacionales, autorizaciones y cambios. Ruta: `/requests`.

## Módulos personalizados

El superadmin puede crear módulos adicionales dinámicamente desde el dashboard. Al crearse, cada módulo hereda automáticamente toda la estructura del sistema: categorías, prioridades, entornos, reglas SLA, FSM de estados, modos de asignación, historial, y visual personalizable. Los módulos creados son accesibles mediante `/{slug-del-módulo}`.

# ESTRUCTURA BASE DE TODOS LOS MÓDULOS

Todos los módulos del sistema, sean core o creados dinámicamente, comparten una estructura visual y funcional consistente. Esto reduce la curva de aprendizaje: un usuario que conoce Helpdesk entiende intuitivamente Inventario y cualquier módulo nuevo.

Cada módulo contiene:

**Header global reutilizado**: el mismo header con notificaciones, calendario, perfil. No cambia entre módulos.

**Sidebar contextual del módulo**: un sidebar lateral izquierdo que es diferente al sidebar global. Sus ítems cambian dinámicamente según el módulo activo, los permisos del usuario y el rol dentro del módulo. La navegación del sidebar del módulo incluye: Resumen (vista principal), Usuarios, Roles, Reportes, Configuración, y Papelera del módulo.

**Banner/cabecera del módulo**: una imagen o banner configurable en la parte superior del área central que muestra el nombre del módulo, un mensaje de bienvenida contextual y una descripción operativa.

**Área operativa central**: donde se renderizan los tickets, solicitudes, tablas, dashboards, métricas y workflows propios del módulo.

# MÓDULO HELPDESK / MESA DE AYUDA (`/helpdesk`)

El módulo Helpdesk es el corazón del sistema de tickets. Define la plantilla base que todos los demás módulos operativos heredan.

## Vista principal

Al ingresar al módulo, el usuario ve la interfaz operativa de tickets. Lo que ve exactamente depende de su rol dentro del módulo:

**El usuario final** (rol `usuario` en el módulo) ve:
- Un botón para crear un nuevo ticket
- La lista de sus tickets propios con estados, prioridades y última actividad
- No ve tickets de otros usuarios
- No ve paneles de supervisión ni métricas de técnicos

**El técnico** (rol `tecnico` en el módulo) ve:
- Su cola de trabajo personal: los tickets asignados a él ordenados por prioridad y fecha
- Tickets colaborativos donde fue invitado a participar
- Un botón/panel estadístico flotante (no permanente) que al abrirlo muestra sus métricas personales: tickets realizados, tiempos promedio, productividad, historial operativo

**El jefe técnico / admin** (roles `jefe_tecnico` o `admin` en el módulo) ve:
- Todos los tickets del módulo
- Panel de técnicos activos y su disponibilidad
- Métricas SLA en tiempo real
- Herramientas de reasignación y supervisión
- Estadísticas completas del módulo

## Cola de trabajo del técnico

El orden en la cola sigue una lógica precisa:
1. Primero aparecen los tickets de días anteriores que quedaron pendientes (por fecha, los más viejos primero)
2. Luego los tickets del día actual ordenados por prioridad: Alta → Media → Baja

Esta lógica garantiza que ningún ticket antiguo quede enterrado bajo trabajo nuevo.

---

## Creación de ticket

El usuario crea un ticket completando los siguientes campos:

**Categoría**: dropdown dinámico. Las categorías son configuradas por el admin del módulo. Ejemplos: Red, Hardware, Software, Cámaras, POS.

**Subcategoría**: se filtra automáticamente según la categoría seleccionada.

**Sede**: la sede donde ocurre el problema. Proviene de la configuración organizacional global.

**Ambiente**: el entorno específico dentro de la sede. Ejemplos: Oficina principal, Sala de servidores, Planta baja.

**Descripción**: texto libre describiendo el problema en detalle.

**Evidencias**: archivos adjuntos opcionales al crear. Formatos permitidos: imágenes (JPG, PNG), PDF, Word, Excel.

Al enviar, el sistema calcula automáticamente la prioridad del ticket según reglas predefinidas (motor PriorityEngine), el tiempo estimado de resolución (motor SLA), y el técnico a quien asignarlo según el modo de asignación configurado en el módulo.

El ticket se crea en estado **ABIERTO**.

---

## Sistema FSM (Finite State Machine) — Ciclo de vida del ticket

El ticket pasa por una serie de estados controlados. No puede saltar estados arbitrariamente. El sistema valida cada transición.

### ABIERTO

Estado inicial. El ticket existe y está en espera de ser atendido. El SLA todavía no ha iniciado.

### EN ESPERA

El técnico asignado abre el ticket y lo revisa. En este estado el técnico puede leer el detalle, ver evidencias y evaluar la complejidad. El SLA todavía NO inicia en este estado. Pasar a EN ESPERA solo indica que el técnico ha visto el ticket.

### EN PROCESO

**El SLA inicia aquí.** Cuando el técnico presiona "Tomar" o "Comenzar", el ticket pasa a EN PROCESO. A partir de este momento:
- El temporizador SLA comienza a correr
- El tiempo se registra en la base de datos
- Las alertas de vencimiento se activan
- El sistema lleva seguimiento de cumplimiento

### REALIZADO

El técnico marca el ticket como resuelto. Para poder pasar a este estado es obligatorio haber adjuntado evidencias de la solución (imágenes, documentos). Sin evidencias el sistema no permite la transición.

Al pasar a REALIZADO ocurre lo siguiente automáticamente:
- El usuario creador del ticket recibe una notificación
- La notificación incluye la solución, las evidencias adjuntas y un token/firma digital
- El usuario tiene 2 días para responder

### CERRADO

Estado final positivo. El ticket queda cerrado cuando:
- El usuario acepta la solución explícitamente, o
- Pasan 2 días sin respuesta del usuario (cierre automático)

Al cerrar, el sistema invita al usuario a calificar la atención recibida: atención, tiempo de respuesta, y calidad de la solución. La calificación es opcional pero alimenta las métricas del módulo.

### REPROCESO

Si el usuario rechaza la solución, debe enviar una justificación y evidencias del problema persistente. El ticket regresa al técnico con estado REPROCESO.

**Regla crítica: solo puede existir 1 reproceso por ticket.** El técnico tiene una segunda oportunidad para resolver el problema.

### ESCALADO

Si el reproceso también falla (el usuario vuelve a rechazar), el sistema escala el ticket automáticamente:
- El jefe técnico toma control del ticket
- Se asigna prioridad máxima (Alta)
- Se genera una reasignación obligatoria
- Se notifica a supervisores

No puede haber un segundo reproceso. El escalamiento es forzoso.

---

## Motor SLA

El SLA no es simplemente un contador de tiempo. Es el núcleo inteligente del módulo:

**Prioridad automática**: basada en combinaciones de categoría, subcategoría, sede, ambiente. Una falla de red en la sede principal en horario pico puede tener prioridad Alta automáticamente aunque el usuario la reporte como "baja". Las reglas se configuran en `/config`.

**Orden dinámico de cola**: los tickets se reordenan automáticamente en la cola del técnico según cambios de prioridad, vencimientos inminentes y escalamientos.

**Asignación inteligente**: considera disponibilidad del técnico, carga laboral actual, horario de trabajo, incapacidades reportadas y especialización técnica.

**Escalamiento automático por vencimiento**: si un ticket está a punto de vencer su SLA, el sistema genera alertas progresivas y puede escalar automáticamente.

**Alertas configurables**: recordatorios previos al vencimiento, notificaciones push al técnico asignado, y alertas al jefe técnico cuando se aproxima el límite.

---

## Workspace del ticket (vista detalle)

Al hacer clic en un ticket, en lugar de abrir un modal se reemplaza la vista principal por un workspace inmersivo de dos paneles sin cambio de URL abrupto. Un botón "volver" regresa a la lista.

**Panel izquierdo (área principal)**:

- **Breadcrumb**: muestra la jerarquía contextual: módulo › categoría › ambiente
- **Título del ticket** y badges de prioridad, estado y SLA (en qué tiempo va)
- **Descripción completa** del problema reportado
- **Meta-datos**: quién lo creó, cuándo, técnico asignado, sede, ambiente
- **Botones de transición FSM**: los botones disponibles cambian según el estado actual. Solo se muestran las transiciones válidas en ese momento.
- **Flujo de confirmación**: algunas transiciones requieren confirmación adicional (por ejemplo, pasar a REALIZADO requiere confirmar que se adjuntaron evidencias)
- **Timeline / historial**: una línea de tiempo completa que muestra cada evento del ticket: creación, asignaciones, cambios de estado, comentarios, adjuntos de evidencias, reuniones programadas, validaciones del usuario, reprocesos y escalamientos. Todo con timestamps y responsables.

**Panel derecho (fondo oscuro #0e2235)**:

- **SLA card**: muestra el deadline del SLA, el tiempo restante o transcurrido, y un badge de estado (en tiempo / próximo a vencer / vencido)
- **Lista de asignados**: quién está trabajando en el ticket con su avatar y datos básicos
- **Invitar colaboradores**: campo de búsqueda para invitar a otro técnico, jefe técnico o supervisor. Requiere especificar el tipo de colaboración: trabajo cooperativo, supervisión, escalamiento técnico, diagnóstico, o validación
- **Sección reuniones**: muestra las reuniones programadas asociadas al ticket (con enlace Meet, fecha, hora). También permite agendar una nueva reunión
- **Botón "Agendar Solución"**: botón verde ubicado al fondo del panel derecho para programar la presentación de la solución al usuario

---

## Sistema de colaboración

Cuando un técnico invita a otro al ticket como trabajo cooperativo, el sistema crea automáticamente un equipo temporal del ticket. Este equipo puede:
- Compartir evidencias y archivos dentro del ticket
- Ver el historial completo conjunto
- Dejar comentarios internos visibles solo para el equipo técnico (no para el usuario final)
- Coordinar la solución de forma conjunta

Los comentarios internos del equipo son diferenciados visualmente de los comentarios/actualizaciones visibles al usuario.

---

## Historial del dispositivo/equipo

El técnico puede consultar el historial completo del equipo o dispositivo asociado al ticket:
- Lista de todos los tickets anteriores relacionados con ese equipo
- Fallas repetidas y sus soluciones previas
- Evidencias de intervenciones anteriores
- Si existe un ticket previo con falla similar, el sistema muestra quién fue el técnico anterior y permite contactarlo directamente desde el ticket actual para coordinar información.

---

## Sistema de reuniones en el ticket

Desde el workspace del ticket se puede programar una reunión con integración preferida a Google Meet:
- Crear reunión: genera enlace automático, registra fecha, hora y motivo
- Los técnicos y admins ven la programación completa, los participantes y el historial de reuniones
- El usuario final solo ve el enlace de reunión generado, la fecha y la hora (no los detalles internos de coordinación)

## Sub-páginas del módulo Helpdesk

### Usuarios del módulo (`/helpdesk/users`)

Muestra únicamente los usuarios que pertenecen al módulo Helpdesk. El acceso al módulo puede estar configurado de dos formas:

**Acceso libre**: cualquier usuario del sistema puede ingresar al módulo sin solicitar permiso.

**Acceso por solicitud** (default): el usuario debe enviar una solicitud de acceso. El admin del módulo o jefe técnico la revisa y aprueba o rechaza. Al aprobarse, el usuario recibe un rol dentro del módulo.

El admin puede desde esta vista: ver todos los miembros actuales, asignarles roles, reasignarlos, suspenderlos del módulo o quitarlos.

### Roles del módulo (`/helpdesk/roles`)

Sistema completo RBAC contextual por módulo. Los roles son independientes: el rol de Juan en Helpdesk no tiene ninguna relación con su rol en Inventario.

**Roles base disponibles**:
- `admin`: administra el módulo completo: usuarios, configuración, reportes, SLA
- `jefe_tecnico`: supervisa técnicos, puede reasignar tickets, ver todas las colas, gestionar SLA
- `tecnico`: atiende tickets, ejecuta trabajo operativo, puede invitar colaboradores
- `usuario`: crea solicitudes/tickets, ve su propio historial

**Roles personalizados**: el admin del módulo puede crear roles adicionales con permisos configurables. Ejemplos: "Técnico Hardware", "Técnico Redes", "Supervisor POS", "Técnico CCTV". Cada rol personalizado tiene su árbol de permisos configurado explícitamente.

Cuando se elimina un rol, el historial y la trazabilidad de acciones pasadas se preservan. Los usuarios afectados simplemente pierden los permisos del rol eliminado a futuro.

### Reportes del módulo (`/helpdesk/reports`)

Métricas exclusivas del módulo:
- Tickets resueltos en el período
- Tiempo promedio de resolución
- Porcentaje de SLA cumplidos (compliance)
- Cantidad de reprocesos y su tasa
- Escalamientos generados
- Productividad por técnico
- Rendimiento técnico individual con promedio de calificaciones

### Configuración del módulo (`/helpdesk/config`)

El admin del módulo configura aquí todos los parámetros operativos:

**Modo de asignación**:
- Manual: el jefe técnico asigna cada ticket explícitamente
- Round Robin: distribución equitativa automática entre técnicos disponibles
- Especialización: el sistema asigna según las categorías que cada técnico tiene habilitadas
- Híbrido: combinación configurable de los modos anteriores

**Modo de prioridad**:
- Automático: el motor SLA calcula la prioridad según las reglas configuradas
- Manual: se permite editar la prioridad. Debe definirse quién puede editarla (solo jefe técnico, o cualquier técnico), y el período durante el cual está permitida la edición manual (fecha inicio y fin).

**Reglas SLA**: qué combinaciones de categoría/subcategoría/sede/ambiente generan qué prioridad y con qué tiempo de resolución.

**Modo de acceso**: si el módulo acepta usuarios libremente o requiere solicitud de acceso.

### Papelera del módulo (`/helpdesk/trash`)

Elementos eliminados dentro del ámbito del módulo Helpdesk: tickets archivados, configuraciones borradas, etc. Funciona igual que la papelera global pero limitada al contexto del módulo.

# MÓDULO GESTIÓN ADMINISTRATIVA (`/requests`)

Es el centro de solicitudes organizacionales del sistema. Funciona como hub de autorizaciones, gestor de cambios administrativos y coordinador de accesos entre módulos.

## Filosofía de funcionamiento

Los usuarios crean solicitudes para cambiar roles, acceder a módulos, corregir datos, reportar problemas de cuenta, etc. El sistema enruta automáticamente cada solicitud al administrador del módulo correspondiente. La solicitud se coordina desde Gestión Administrativa, pero los cambios reales se ejecutan en el módulo correspondiente. Por ejemplo: si alguien solicita un cambio de rol en Inventario, la solicitud aparece en Gestión Administrativa, el admin de Inventario la revisa y la ejecuta, pero el cambio real de rol ocurre dentro del módulo Inventario porque ese módulo controla sus propios roles y usuarios.

Todos los usuarios del sistema tienen acceso a este módulo para ver sus propias solicitudes. Solo superadmin y admins ven las solicitudes de todos los usuarios.

## Vista principal

Al ingresar, el usuario ve la lista de sus solicitudes activas e históricas. El admin ve todas las solicitudes del sistema o filtradas por módulo. Cada solicitud en la lista muestra: tipo, descripción breve, módulo relacionado, prioridad, estado y fecha de creación.

## Crear una solicitud

Al presionar el botón de crear solicitud se abre un modal contextual (`CreateRequestModal`) con los siguientes campos:

**Tipo de solicitud**: dropdown con los tipos disponibles:

- **Cambio de rol**: el usuario solicita que su rol dentro de un módulo sea modificado
- **Acceso a módulo**: el usuario solicita ser incluido en un módulo al que aún no tiene acceso
- **Ajuste de permisos**: solicitud de modificación de permisos específicos
- **Problema de cuenta**: reportar un problema con el acceso, contraseña u otra situación de cuenta
- **Reactivación**: solicitar reactivación de una cuenta suspendida
- **Revocación de acceso**: solicitar que se le retire acceso a alguien (o a uno mismo)
- **Traslado de usuarios**: solicitar traslado entre módulos, sedes o áreas
- **Problema técnico**: problemas técnicos que no corresponden a un módulo específico
- **Corrección de datos**: solicitar corrección de información organizacional incorrecta
- **Tarea**: solicitud de tarea interna
- **Otros**: para casos que no encajan en los anteriores. Al seleccionar esta opción, el sistema muestra una explicación contextual de qué casos aplican y las limitaciones del tipo.

**Módulo relacionado**: dropdown dinámico con todos los módulos del sistema. Proviene de la base de datos en tiempo real.

**Descripción**: campo de texto libre para detallar la solicitud.

**Prioridad**: en la mayoría de tipos de solicitud, la prioridad es calculada automáticamente por el sistema (motor PriorityEngine) según el tipo de solicitud, el cargo del usuario y el contexto organizacional. Solo el tipo "Otros" permite que el usuario indique la prioridad manualmente. Esto evita que todos los usuarios marquen todo como "urgente".

## Prioridad automática y SLA

Al crear la solicitud, el sistema:
1. Calcula la prioridad según reglas en `config.priority_rules`: considera el tipo de solicitud y el nivel del cargo del usuario (position_level). Un cargo gerencial puede elevar automáticamente la prioridad.
2. Calcula el deadline SLA según `config.sla_rules`: crítica=2h, alta=8h, media=24h, baja=72h (valores configurables).
3. Resuelve el destinatario: busca al usuario con rol `admin` en el módulo indicado. Si no hay admin asignado, la solicitud queda escalada automáticamente al superadmin.

## Flujo de aprobación

1. Admin del módulo recibe notificación en el header (bell icon)
2. Admin revisa la solicitud en la bandeja de Gestión Administrativa
3. Admin puede: aprobar, rechazar, o escalar a nivel superior
4. Si aprueba: ejecuta el cambio en el módulo correspondiente
5. El usuario recibe notificación del resultado

## Escalamiento de solicitudes

Los admins de módulo pueden escalar solicitudes cuando la acción supera sus permisos, afecta múltiples módulos, involucra seguridad organizacional o requiere aprobación de nivel superior. Las solicitudes escaladas llegan al superadmin.

## Modales de gestión de solicitudes

**`CancelRequestModal`**: permite al usuario cancelar su propia solicitud si aún no ha sido procesada. Requiere confirmar la cancelación.

**`EscalateRequestModal`**: usado por admins para escalar la solicitud a nivel superior. Requiere indicar el motivo del escalamiento.

**`ExecuteRequestModal`**: usado por el admin para aprobar y ejecutar la solicitud. Muestra el resumen de la acción a ejecutar y requiere confirmación.

## Sub-páginas del módulo

Las sub-páginas de Gestión Administrativa (`/requests/users`, `/requests/roles`, `/requests/reports`, `/requests/trash`) son visibles únicamente para el admin y superadmin. La sub-página `/requests/config` redirige directamente a la configuración global `/config` ya que Gestión Administrativa comparte la configuración organizacional global.

## Trazabilidad obligatoria

Toda solicitud conserva: historial completo de cambios, responsables en cada etapa, aprobaciones y rechazos, escalamientos, comentarios internos, y timestamps precisos de cada acción.

# MÓDULOS DINÁMICOS (`/{module-slug}`)

Los módulos creados por el superadmin desde el dashboard son accesibles mediante su slug único. Por ejemplo, si el superadmin crea un módulo llamado "Gestión Comercial", su ruta sería `/gestion-comercial`.

## Comportamiento idéntico al Helpdesk

Todos los módulos dinámicos heredan exactamente la misma estructura visual y funcional del Helpdesk: tickets, FSM, SLA, asignaciones, cola de técnicos, workspace, colaboración, reuniones, calificaciones. Por defecto, todos los módulos nuevos requieren solicitud de acceso (`access_mode = request`). El usuario solicita acceso via Gestión Administrativa. El admin del módulo aprueba y asigna un rol.

Si el admin del módulo cambia a modo acceso libre (`access_mode = open`), cualquier usuario puede ingresar sin necesidad de solicitud.

## Configuración del módulo dinámico (`/{slug}/config`)

El admin del módulo puede configurar:

**Modo de asignación**: manual, round robin, especialización o híbrido.

**Modo de prioridad**: automático (por reglas) o manual. Si es manual, se especifica quién puede editar la prioridad y durante qué período.

**Categorías y subcategorías**: definidas específicamente para ese módulo.

**Entornos**: los ambientes o contextos dentro del módulo.

**Reglas SLA**: combinaciones categoría/prioridad con tiempos de resolución.

# CALENDARIO OPERATIVO GLOBAL (`/calendar`)

El calendario no es una funcionalidad secundaria. Es un núcleo operativo transversal conectado con módulos, tickets, SLA, tareas, usuarios, reuniones, mantenimientos y automatizaciones.

## Acceso

El calendario no vive dentro del sidebar global. Se accede desde el header mediante un botón contextual que puede mostrar un dropdown rápido o un panel lateral expandible. El objetivo es acceso instantáneo con mínimo número de clics, disponible desde cualquier pantalla del sistema.

## Layout del calendario

El calendario tiene un diseño two-panel:

**Panel principal**: ocupa la mayor parte del espacio. Contiene el sub-header con contexto del módulo activo, el switcher de vistas, la tira de estadísticas (chips), los filtros y la vista del calendario propiamente dicha.

**Panel derecho** (296px fijo): muestra los eventos del día seleccionado, el panel de disponibilidad de usuarios del módulo activo, y el botón de crear evento.

## Vistas disponibles

**Vista Mes**: implementación personalizada con un grid de días. Cada día con eventos muestra puntos de colores indicadores. Al hacer clic en un día, el panel derecho actualiza la lista de eventos de ese día. El día de hoy tiene un highlight especial. El día seleccionado tiene un círculo coral (`#ff5e3a`).

**Vista Semana**: powered by FullCalendar, muestra los eventos distribuidos en la semana con bloques de hora.

**Vista Día**: powered by FullCalendar, detalle por hora del día seleccionado.

**Vista Agenda**: powered by FullCalendar, lista lineal de tareas, eventos y seguimientos próximos.

## Sistema de colores de puntos (month grid)

Los puntos en el grid de mes tienen colores que representan el origen del evento:
- **Coral (`#ff5e3a`)**: solicitudes y tareas de Gestión Administrativa
- **Naranja (`#f59e0b`)**: deadlines de SLA de tickets activos
- **Morado (`#8b5cf6`)**: eventos creados explícitamente via calendario (CRUD)

## Tira de estadísticas

Una barra de chips en la parte superior del calendario muestra contadores en tiempo real: Total eventos / Pendientes / En proceso / Completados / Aprobados / SLA activos / Eventos calendario.

## Filtros del calendario

Cuatro selectores de filtro: Estado, Prioridad, Tipo, Origen. Los filtros aplican tanto en el cliente (respuesta inmediata) como en el servidor (data actualizada).

## Panel de disponibilidad

El panel derecho muestra la lista de usuarios activos en el módulo activo con colores según su rol. Permite visualizar rápidamente quién está disponible para asignar reuniones o tareas.

## Creación de eventos

Al presionar el botón de crear evento en el panel derecho se abre el modal `CreateEventModal`. Permite crear:

**Eventos personales**: solo visibles para el creador. No afectan SLA, no generan automatizaciones globales, no aparecen en métricas compartidas. Uso: recordatorios, reuniones personales, tareas privadas.

**Eventos de módulo**: asociados a un módulo específico. Visibles según rol y permisos en ese módulo. Ejemplos: soporte programado, revisiones técnicas, auditorías SLA, mantenimientos preventivos, aprobaciones.

**Eventos globales**: visibles para toda la organización o roles definidos. Solo admins/superadmin pueden crearlos. Ejemplos: mantenimiento del sistema, reuniones corporativas, capacitaciones, eventos organizacionales.

## Detalle de evento

Al hacer clic en un evento en el panel derecho se abre el `EventDetailPopup` con: título, descripción, fecha/hora, tipo, módulo asociado, participantes, estado, prioridad, y acciones disponibles según permisos (editar, cancelar, cambiar estado).

## Integración SLA en el calendario

Los tickets con SLA activo aparecen automáticamente como eventos en el calendario. En el panel derecho, los tickets SLA se muestran por encima de las solicitudes (urgency-first). Cada ticket SLA muestra el deadline, el estado del SLA y permite abrir el detalle del ticket. Al hacer clic se abre `TicketSlaPopup` que es solo lectura.

## Eventos recurrentes

El sistema soporta recurrencia: diaria, semanal, mensual, personalizada. Útil para reuniones semanales de equipo, mantenimientos mensuales, auditorías periódicas.

## Audit log del calendario

Una barra de 72px al final del panel muestra un log en tiempo real de las últimas acciones del calendario (creación, edición, cancelación de eventos). Scroll automático al último evento. Fuente monospace.

## Comportamiento según rol

**Usuario estándar**: puede visualizar el calendario, ver sus tareas asignadas y gestionar sus eventos personales. No puede crear eventos globales ni modificar eventos de módulo.

**Técnico**: puede ver tareas técnicas, gestionar sus asignaciones, programar seguimientos y asociar reuniones a tickets.

**Jefe técnico / Supervisor**: puede ver el calendario del equipo, reasignar tareas, coordinar disponibilidad y supervisar SLA.

**Admin de módulo**: puede crear eventos del módulo, gestionar tareas colectivas, programar mantenimientos y coordinar reuniones operativas.

**Superadmin**: acceso completo. Puede ver todos los eventos, filtrar globalmente, crear eventos globales y gestionar disponibilidad organizacional.

# GESTIÓN GLOBAL DE ROLES (`/roles`)

Uno de los núcleos críticos de la plataforma. Desde aquí se administran los roles globales del sistema y sus permisos.

## Tipos de roles

**Roles globales**: definen el nivel de acceso base al sistema. Solo existen dos: `superadmin` (acceso total) y `usuario` (acceso base).

**Roles por módulo**: independientes en cada módulo. Juan puede ser técnico en Helpdesk y admin en Inventario simultáneamente. Estos se gestionan dentro de cada módulo en sus respectivas sub-páginas de roles, no aquí.

## Vista de permisos

Los permisos de cada rol se muestran en formato árbol jerárquico. Cada nodo del árbol es un permiso con nombre descriptivo.

**Regla crítica**: habilitar un permiso padre NO habilita automáticamente sus permisos hijos. Todo permiso debe activarse explícitamente. Esto da control granular total sobre qué puede hacer cada rol.

Los permisos pueden controlar: visualización de secciones, acceso a rutas, creación de recursos, edición, eliminación, restauración desde papelera, exportación de reportes, gestión de configuración, acceso en modo mantenimiento, y administración.

## Operaciones sobre roles

**Crear rol**: define nombre, descripción y asigna permisos desde el árbol.

**Editar rol**: modifica nombre, descripción y toggle individual de cada permiso.

**Grant/revoke subtree**: permite activar o desactivar todos los permisos hijos de un nodo padre con un solo clic. Útil para dar acceso rápido a una sección completa.

**Activar/desactivar rol**: deshabilita el rol sin eliminarlo.

**Eliminar rol**: el historial y trazabilidad de acciones pasadas se preserva. Los usuarios que tenían ese rol simplemente pierden los permisos a futuro.

## Retención automática

Los elementos permanecen 90 días en papelera. Después de ese período son eliminados definitivamente de forma automática por el sistema.

# SISTEMA DE ROLES POR MÓDULO — PRINCIPIOS

## Independencia completa

Los roles son completamente independientes entre módulos. El hecho de que Juan sea admin en Helpdesk no le da ningún privilegio en Inventario. Cada módulo gestiona sus propios roles, usuarios y permisos de forma autónoma.

## Multirol simultáneo

Un usuario puede tener múltiples roles al mismo tiempo en múltiples módulos:

Ejemplo:
- Rol global: usuario (base)
- Helpdesk: técnico + jefe_tecnico (puede ser ambos)
- Inventario: admin
- Módulo Comercial: técnico

## Principio de responsabilidad modular

Cada módulo es responsable de administrar sus roles, usuarios internos, permisos contextuales y configuraciones operativas. Gestión Administrativa únicamente centraliza solicitudes, enruta procesos, mantiene trazabilidad y coordina aprobaciones. Los cambios reales los ejecuta cada módulo en su propio contexto.

# MOTOR RBAC — 60 PERMISOS

El sistema tiene un motor de permisos completo con 60 permisos organizados por scope:

**Scope `global` (31 permisos)**: controlan acceso a secciones administrativas globales: usuarios, roles, módulos, configuración, reportes, papelera, y operaciones globales.

**Scope `gestion` (17 permisos)**: controlan operaciones dentro de Gestión Administrativa: ver solicitudes, aprobarlas, escalarlas, gestionar usuarios del módulo, roles, reportes.

**Scope `helpdesk` (8 permisos)**: operaciones de Helpdesk: ver tickets, crear, asignar, escalar, cerrar, acceder a SLA, gestionar técnicos.

**Scope `inventario` (4 permisos)**: operaciones básicas de inventario.

## Formato de clave de permiso

Todos los permisos siguen el formato: `{scope}:{sección}:{acción}`

Ejemplos: `global:users:edit`, `helpdesk:tickets:assign`, `gestion:requests:approve`

## Fast-path del superadmin

El superadmin tiene `is_superadmin = true`. El store de permisos recibe `['*']` lo que significa que `hasPermission()` siempre retorna true para cualquier permiso. El superadmin nunca es bloqueado por ninguna verificación de permisos.

## Componentes de verificación

`PermissionGate` en el frontend: componente que envuelve secciones de UI. Si el usuario no tiene el permiso, el contenido no se renderiza. Ejemplo: `<PermissionGate perm="global:users:create">` oculta un botón de crear usuario para quien no tiene ese permiso.

`usePermission(key)`: hook que retorna true/false para usar en lógica condicional dentro de componentes.

# PRESENCIA ONLINE Y GEOLOCALIZACIÓN

## Sistema de heartbeat

El sistema envía automáticamente una señal silenciosa al servidor cada 60 segundos indicando que el usuario sigue activo. Este heartbeat actualiza `last_seen_at` en el perfil del usuario. Si la última señal fue hace menos de 2 minutos, el usuario se considera online.

## Geolocalización de sesiones

Al hacer login, el sistema realiza en segundo plano una consulta a ip-api.com con la IP del cliente. Esto no bloquea el login. Una vez obtenida la información geográfica, se almacena en la sesión: ciudad, país, código de país, latitud y longitud. Esta información es visible en la pestaña de sesiones del perfil del usuario, con bandera emoji del país, nombre de la ciudad, y un mapa miniatura OSM.

# NOTIFICACIONES

Las notificaciones son internas (in-app) como canal principal. El bell del header muestra el conteo de no leídas. Al hacer clic se despliega la lista de notificaciones con opciones de marcar como leída individualmente o marcar todas como leídas de una vez.

Las notificaciones se generan automáticamente en los siguientes eventos:
- Transición de estado de un ticket (el usuario recibe cuando su ticket cambia de estado)
- Solución lista para validar (usuario recibe evidencias + solicitud de aprobación)
- Aprobación o rechazo de solicitud en Gestión Administrativa
- Invitación a colaborar en un ticket
- Alerta de vencimiento de SLA
- Evento del calendario asignado o próximo
- Resultado de acciones administrativas sobre la propia cuenta

Adicionalmente, las notificaciones se pueden enviar por correo electrónico via Resend API (Gmail por defecto).

# REGLAS INVARIABLES DEL SISTEMA

Las siguientes reglas nunca deben ser violadas, independientemente de los cambios que se hagan al sistema:

1. **Nunca auto-crear usuarios**. Ni por OAuth ni por ningún otro flujo. Toda cuenta requiere creación previa por un administrador.

2. **Nunca eliminar directamente desde tablas principales**. Todo pasa por soft-delete a la papelera.

3. **Nunca revelar información sensible en mensajes de error**. Si el email existe, si la contraseña es incorrecta, qué campo falló: ninguno de estos datos debe comunicarse al usuario.

4. **Las rutas de módulos dependen del slug del módulo**: `/{slug}`, nunca `/tickets` como ruta genérica del sistema.

5. **Los módulos core son ineliminables**: Helpdesk, Inventario y Gestión Administrativa no pueden ser eliminados bajo ninguna circunstancia.

6. **Solo el superadmin crea módulos nuevos**. Los admins de módulo administran los módulos que tienen asignados pero no pueden crear nuevos.

7. **El SLA inicia en EN PROCESO, no en EN ESPERA**. Pasar a EN ESPERA es solo una revisión; el temporizador real empieza cuando el técnico toma el ticket.

8. **Máximo 1 reproceso por ticket**. Al segundo fallo, el escalamiento es obligatorio e irreversible.

9. **Los roles son independientes por módulo**. El rol de un usuario en un módulo no afecta su rol en otro módulo.

10. **Los permisos son siempre explícitos**. Activar un permiso padre no activa automáticamente ningún permiso hijo. Todo debe definirse por separado.

11. **La información organizacional (sedes, áreas, cargos) siempre proviene de la DB**. Nunca debe hardcodearse en formularios. Los dropdowns deben poblar dinámicamente desde la configuración real del sistema.

12. **Los elementos en papelera se conservan 90 días**. La eliminación antes del vencimiento requiere re-autenticación con contraseña.

13. **El superadmin bypasea toda verificación de permisos**. `is_superadmin = true` es un fast-path que no puede ser bloqueado por ningún guard de permiso.

14. **La estructura visual actual no se modifica sin instrucción explícita**. El enfoque prioritario es funcionalidad, permisos, trazabilidad y lógica operativa. El rediseño visual es una decisión consciente, no un efecto secundario.
