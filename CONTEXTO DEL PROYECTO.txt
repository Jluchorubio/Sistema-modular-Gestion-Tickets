
# CONTEXTO DEFINITIVO — ARQUITECTURA GENERAL DEL SISTEMA MODULAR OPERACIONAL

# VISIÓN GENERAL DEL PROYECTO

El sistema es una plataforma modular empresarial orientada a:

* Gestión de tickets
* Gestión administrativa
* Inventario
* Automatización operacional
* SLA inteligentes
* Gestión organizacional
* Calendario operacional
* Reportes
* Auditoría
* Roles y permisos

La arquitectura está diseñada para que:

* Existan módulos independientes
* Cada módulo pueda operar de manera autónoma
* Los módulos compartan:

  * usuarios
  * autenticación
  * permisos
  * sistema de notificaciones
  * calendario
  * motor organizacional
  * automatizaciones

El sistema es SINGLE TENANT.

Inicialmente:

* existe una sola empresa
* una sola instancia
* una sola organización

Pero la arquitectura debe permitir escalar posteriormente.

# ARQUITECTURA MODULAR

Cada módulo es independiente.

Ejemplo:

* Helpdesk
* Inventario
* Gestión Administrativa
*  

  Otros módulos futuros

Estos no se toman como modulos:

* Calendario
* Reportes

* hereda la plantilla operacional del modulo Helpdesk
* genera rutas automáticamente

* queda completamente desacoplado

* puede personalizarse completamente

# SISTEMA DE ROLES Y PERMISOS — CONTEXTO FUNCIONAL Y ARQUITECTÓNICO

El sistema maneja un RBAC (Role Based Access Control) modular y dinámico.

NO es un sistema simple de:
“admin / usuario”.

La arquitectura está diseñada para soportar:

* permisos globales
* permisos por módulo
* permisos dinámicos
* múltiples roles
* múltiples contextos operativos
* escalabilidad modular

---

# OBJETIVO DEL SISTEMA DE ROLES

El sistema debe permitir controlar:

* qué puede ver un usuario
* qué puede crear
* qué puede editar
* qué puede eliminar
* qué módulos puede usar
* qué acciones administrativas puede ejecutar
* qué vistas operativas puede visualizar

Todo esto:
sin hardcodear permisos manualmente en frontend.

---

# ESTRUCTURA GENERAL

El sistema maneja:

## 1. Roles globales

Aplican a TODO el sistema.

Ejemplos:

* superadmin
* usuario

Estos determinan:

* acceso global
* administración general
* acceso transversal

---

## 2. Roles por módulo

Cada módulo puede tener sus propios roles independientes.

Ejemplo:

Módulo Helpdesk:

* técnico
* coordinador
* supervisor

Módulo Inventario:

* almacenista
* auditor inventario

Módulo Gestión Administrativa:

* aprobador
* gestor solicitudes

Los roles de un módulo NO afectan automáticamente a otro módulo.

---

# SUPERADMIN

El rol superadmin:

* SIEMPRE tiene acceso total
* NO necesita asignarse permisos manualmente
* NO depende del árbol de permisos
* bypass completo de validaciones RBAC

Esto evita:

* bloquear accidentalmente al superadmin
* perder acceso administrativo crítico

El superadmin puede:

* ver todos los módulos
* administrar usuarios
* administrar roles
* crear módulos
* editar configuración global
* editar automatizaciones
* ver auditorías
* acceder a todos los endpoints protegidos

---

# PERMISOS

Los permisos NO son simples strings sueltos.

El sistema maneja permisos estructurados jerárquicamente.

Ejemplo:

```txt
tickets
 ├── view
 ├── create
 ├── edit
 ├── assign
 ├── close
 └── delete
```

o:

```txt
inventory
 ├── assets.view
 ├── assets.create
 ├── assets.edit
 ├── assets.delete
 ├── qr.generate
 └── reports.view
```

---

# IMPORTANTE — PADRE NO HEREDA HIJOS

Regla arquitectónica IMPORTANTÍSIMA:

Activar un permiso padre
NO activa automáticamente los hijos.

Ejemplo:

```txt
tickets
```

NO concede automáticamente:

```txt
tickets.delete
tickets.assign
tickets.close
```

Esto es intencional.

Porque en sistemas empresariales:
las acciones críticas deben concederse explícitamente.

---

# HERENCIA VISUAL ≠ HERENCIA REAL

La UI puede mostrar jerarquía visual:

```txt
Tickets
 ├── Ver
 ├── Crear
 ├── Eliminar
```

pero internamente:
cada permiso es independiente.

Esto evita:

* escalaciones accidentales
* permisos excesivos
* errores administrativos

---

# CREACIÓN DINÁMICA DE PERMISOS

Cuando el superadmin crea un nuevo módulo:

el sistema automáticamente genera:

* namespace del módulo
* endpoints
* rutas
* permisos base
* roles iniciales opcionales

Ejemplo:

Nuevo módulo:

```txt
Mantenimiento
```

Genera automáticamente:

```txt
maintenance.view
maintenance.create
maintenance.edit
maintenance.delete
maintenance.config
maintenance.reports
```

Y estos aparecen inmediatamente en el módulo de Roles.

---

# LOS PERMISOS SON DINÁMICOS

NO quiero permisos hardcodeados en frontend.

Los permisos deben obtenerse desde backend/database.

Porque:

* los módulos son dinámicos
* los permisos pueden crecer
* los roles cambian
* el sistema es escalable

Frontend debe renderizar:

* sidebar
* botones
* tabs
* acciones
* rutas

según permisos reales del usuario.

---

# COMPORTAMIENTO DEL SIDEBAR SEGÚN PERMISOS

El sidebar cambia dinámicamente.

Ejemplo:

Si el usuario NO tiene:

```txt
reports.view
```

```txt

```

```txt

```

```txt

```

* globalmente
* como dentro de módulos

# PERMISSION GUARDS

Backend:

Todos los endpoints sensibles deben usar:

* JWT Guard
* Permission Guard
* Role validation
* Module access validation

Frontend:

* PermissionGate
* PermissionGuard
* usePermission()
* useModuleAccess()

Para ocultar:

* vistas
* botones
* acciones
* configuraciones

sin depender solo del backend.

## 1. Colaboración entre técnicos

Un técnico podrá:

* Solicitar apoyo de otro técnico
* Crear una videollamada colaborativa
* Compartir contexto del ticket
* Resolver incidentes conjuntamente

Especialmente útil para:

* Tickets críticos
* Casos especializados
* Reprocesos
* Escalamientos

---

## 2. Atención directa al usuario (evaluación futura)

Opcionalmente, el sistema podrá permitir:

* Videollamadas entre técnico y usuario final

Pero esto debe evaluarse cuidadosamente porque:

* Puede aumentar carga operativa
* Puede romper flujo asincrónico del sistema
* No todos los tickets requieren contacto directo

Por ahora, debe diseñarse como:

* Funcionalidad opcional
* Activable por módulo o política
* Controlada por permisos
