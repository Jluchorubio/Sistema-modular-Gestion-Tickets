# SISTEMA DE ROLES Y PERMISOS

El sistema maneja un RBAC (Role Based Access Control) modular y dinámico.

NO es un sistema simple de "admin / usuario".

La arquitectura soporta:

* permisos globales
* permisos por módulo
* permisos dinámicos
* múltiples roles
* múltiples contextos operativos
* escalabilidad modular

## Estado actual del RBAC

60 permisos definidos en DB (global:31, gestion:17, helpdesk:8, inventario:4).

PermissionGate implementado en frontend.

Guards en backend: JwtAuthGuard + PermissionGuard + RolesGuard.
Todos los controllers cubiertos con @RequirePermission().

Grants por rol:
* superadmin: 60 (bypass directo vía is_superadmin=true, no consulta grants)
* usuario: 12 (acceso básico + requests create/view + tickets create/view + comments + inventario view)

Sidebar:
* HELPDESK_NAV items: filtrados por allowedRoles
* GESTION_NAV items: filtrados por permKey vía usePermissionsStore
* superadmin bypassa ambos checks

Invalidación de caché:
* Backend: cache 60s TTL con invalidateUser() / invalidateAll()
* Frontend: invalidateQueries(['permissions-mine']) post-cambio roles

El superadmin:

* SIEMPRE tiene acceso total
* NO necesita asignarse permisos manualmente
* bypass completo de validaciones RBAC

# PAPELERA

Ruta: /trash

## Reglas

Soft delete:

* 90 días

Luego:

* Hard delete automático

Para eliminar manualmente antes de los 90 días:

* el usuario debe colocar su contraseña
* solo superadmin o admin pueden eliminar
* superadmin: acceso a todo lo global y del admin
* admin: solo lo del módulo que le corresponde
* se pueden seleccionar muchos a la vez (bulk)

Antes del hard delete el sistema:

* notifica progresivamente
* muestra alertas de expiración próxima

Los tickets son registros permanentes del sistema.

NO se eliminan, ni siquiera con soft delete.

* Cancelar
* Archivar

* trazabilidad

# TRANSITION RULES — FSM CON CONTROL DE ROL

La tabla tickets.transitions tiene columna allowed_roles text[].

Esta columna define qué roles pueden ejecutar cada transición.

El backend valida el rol del actor ANTES de aplicar el cambio de estado.

Si el rol no está permitido: ForbiddenException 403.

Superadmin siempre bypassa.

Matriz de roles por transición:

* Tomar ticket (abierto → en_proceso): tecnico, jefe_tecnico, admin_modulo
* Reanudar (en_espera → en_proceso): tecnico, jefe_tecnico, admin_modulo
* Solicitar información (en_proceso → en_espera): tecnico, jefe_tecnico, admin_modulo
* Marcar realizado (en_proceso → realizado): tecnico, jefe_tecnico, admin_modulo
* Rechazar (en_proceso → rechazado): jefe_tecnico, admin_modulo
* Marcar resuelto (en_proceso → resuelto): tecnico, jefe_tecnico, admin_modulo
* Aprobar y cerrar (realizado → cerrado): usuario, jefe_tecnico, admin_modulo
* Rechazar solución (realizado → reproceso): usuario, jefe_tecnico, admin_modulo
* Retomar para reproceso (reproceso → en_proceso): tecnico, jefe_tecnico, admin_modulo
* Cerrar (resuelto → cerrado): jefe_tecnico, admin_modulo
* Reabrir (resuelto → en_proceso): jefe_tecnico, admin_modulo

# VALIDACIÓN DE CIERRE Y CALIFICACIÓN

Cuando el técnico marca el ticket como "Realizado":

* el usuario creador recibe notificación
* el usuario ve banner de validación con firma digital
* puede Aprobar (con firma) o Rechazar (con justificación)

Cuando el usuario aprueba:

* el ticket pasa a estado final
* el sistema muestra banner de calificación (5 estrellas + comentario opcional)
* la calificación se asocia al técnico asignado (owner)

Cuando el usuario rechaza:

* el ticket va a Reproceso
* no se solicita calificación

Si el ticket se cierra automáticamente por timeout (sin respuesta):

* NO se solicita calificación

Regla crítica:

* Solo 1 reproceso permitido
* Si falla el reproceso: escala automáticamente

# SLA DEL HELPDESK

El SLA es un motor operacional inteligente.

NO solo mide tiempo.

Controla:

* prioridad
* orden de atención
* escalación
* pausas
* vencimientos
* horarios laborales
* feriados

El reloj SLA:

* inicia cuando el ticket pasa a "En proceso"
* se pausa automáticamente en "En espera"
* se reanuda (y extiende el deadline) al salir de En espera
* se cierra como "met" o "breached" al llegar a estado final

# REGLAS ARQUITECTÓNICAS CRÍTICAS

## Tickets

Los tickets son registros permanentes.

NO se eliminan.

NO van a papelera.

Se Cancelan o Archivan.

## Roles

Los 60 permisos RBAC actuales son intocables.

Nuevos módulos generan sus propios permisos en un namespace propio.

## Módulos

Solo el superadmin puede crear módulos.

Los módulos built-in (Helpdesk, Inventario) no pueden eliminarse.

## Config

/config solo tiene 4 secciones: General, Sedes, SLA-Tickets, Calendario.

Importar usuarios va en /users.

Módulo/users filtra por módulo asignado.

## Sidebar

El sidebar global SOLO para superadmin y roles con permisos globales.

El sidebar modular para roles con acceso al módulo.

Usuarios normales no ven sidebar global.

# INTEGRACIÓN DE VIDEOLLAMADAS

Dentro del ticket se puede:

* crear reunión
* generar enlace
* invitar participantes
* registrar participación, fecha, duración

Integración actual: Google Meet API + Google Calendar API.

La implementación es modular para permitir reemplazar proveedor posteriormente.

Proveedores soportados:
* Google Meet — genera link automáticamente
* Microsoft Teams — NO genera link automático. El usuario debe pegar el link manualmente.
* Zoom — NO genera link automático. El usuario debe pegar el link manualmente.

La UI debe indicar claramente cuando el proveedor requiere link manual.

Usos:

* Colaboración entre técnicos
* Atención directa al usuario (opcional, activable por módulo)

Las reuniones quedan en el timeline del ticket y se reflejan en el calendario del sistema.

# VALIDACIONES CRÍTICAS DEL SISTEMA

## Usuarios

* Edad mínima: 16 años (validado en birthdate frontend + backend)
* Birthdate: no fechas futuras, no antes de 1900
* Múltiples sesiones simultáneas: SÍ permitidas
* Sesiones activas: mostrar ubicación aproximada (ciudad/país), NO mostrar IP real
* Contraseña: mínimo 8 caracteres, una mayúscula, una minúscula, un número

## Complete Profile

* La contraseña temporal por defecto es Ticket2026!
* Al ingresar por primera vez el usuario debe completar perfil + cambiar contraseña
* Dropdowns dinámicos: empresa → departamento → área → bloque → ambiente (cascading desde org.nodes)
* Si la DB no tiene org.nodes poblados, los dropdowns estarán vacíos — es comportamiento correcto

## Contraseña — política backend

* min 8 caracteres
* al menos 1 mayúscula
* al menos 1 minúscula
* al menos 1 número

La validación frontend (securitySchema) debe ser IDÉNTICA a la política backend.

## Tickets

* NO se eliminan ni van a papelera
* Solo se Cancelan o Archivan
* Preserva trazabilidad, SLA histórico, métricas, auditoría
